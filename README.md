# Lok Lagbe? | লোক লাগবে?
### Full-Stack Service Marketplace
A home service marketplace for Bangladesh where customers hire verified professionals for plumbing, electrical, cleaning, and more.
## Tech Stack
- **Frontend**: Next.js 14 (App Router) + TypeScript + Tailwind CSS
- **Backend**: Next.js API Routes + Supabase
- **Database**: PostgreSQL (Supabase) with RLS
- **Auth**: Supabase Auth (JWT-based)
## Setup
### 1. Install
```bash
npm install
```
### 2. Supabase
- Create project at supabase.com
- Run `supabase/schema.sql` in SQL Editor
- Copy URL and anon key
### 3. Environment
```bash
cp .env.local.example .env.local
# Fill in your Supabase credentials
```
### 4. Create Admin
- Sign up at /signup
- In Supabase → profiles table → set your role to 'admin'
### 5. Run
```bash
npm run dev
```
## Roles
| Role | Dashboard | Access |
|------|-----------|--------|
| Admin | /admin | All users, orders, revenue |
| Vendor | /vendor | Add services, manage jobs |
| User | /user | Browse, book, order history |
## Database ERD

- **profiles** → has many **orders** (as customer)
- **profiles** (role = vendor) → has many **services**
- **categories** → has many **services**
- **services** → has many **orders**
- **orders** → references one **profile** (customer) and one **service**

## Vibe Coding Workflow

**Tools used:** Claude

**Process:** I started with a single detailed base prompt describing the full
product — roles, database structure, core user flows — to get Claude oriented
on the overall system before touching any code. From there I worked feature
by feature (auth/RBAC, vendor dashboard, service catalog, checkout flow),
giving Claude the existing schema and relevant files as context for each new
feature so it stayed consistent with what was already built, rather than
prompting for the whole app in one shot.

### Where it succeeded
- **Supabase setup** — project structure, auth configuration, and the initial
  schema came together correctly with minimal back-and-forth.
- **UI scaffolding** — Tailwind layouts and component structure for the
  dashboards were usable close to first pass.

### Where it failed and required manual fixes
The **order/payment workflow** needed the most hand intervention. Claude's
first version had several logic issues that weren't obvious until I traced
through the flow:
- On successful payment, the order notification fired to **both admin and
  vendor simultaneously** — not a practical flow, since the vendor shouldn't
  be notified to act on an order before it's actually confirmed/assigned.
- **Cancellations didn't trigger a refund** — money stayed with the vendor
  even when an order was cancelled, with no reversal logic.
- **Funds were released to the vendor immediately on payment**, before the
  job was actually performed — meaning a vendor could get paid for a job
  they never completed.

The root cause was that Claude treated "payment succeeded" as the end of the
flow rather than modeling the order as a proper lifecycle with distinct
states (paid → in progress → completed → funds released, with cancellation
able to branch off at any pre-completion state). It wired the happy path
correctly but didn't account for timing and reversibility. I rewrote the
order state transitions and payment-release logic by hand so that funds are
held until job completion is confirmed, cancellations trigger a refund path,
and notifications are scoped to the correct role at each stage.

Separately, **RLS (Row Level Security) policies** generated by Claude needed
manual correction — the initial policies were either too permissive (allowing
cross-role reads they shouldn't) or too restrictive (blocking legitimate
queries), so I went through each table's policy by hand to match the actual
access rules each role should have.
